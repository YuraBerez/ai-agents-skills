# Skill: PR Check

Review a pull request against its Technical Task: verify acceptance criteria coverage, flag scope creep, missing tests, and broken links.

## Trigger

Invoked via `/pr-check`. The user must provide:
1. The PR description or a link to the PR
2. The Technical Task (from `/idea-to-task`) it relates to

If either is missing, ask for it before proceeding.

## Process

### Step 1 — Parse inputs

From the PR description extract:
- Changes made (grouped by area)
- Acceptance criteria coverage table (if present)
- "Out of Scope" section
- Notes for Reviewer

From the Technical Task extract:
- All acceptance criteria
- Definition of Done checklist
- Testing Strategy
- Out of Scope
- Open Technical Questions

### Step 2 — Ask clarifying questions

Ask only if needed:

- **Access to diff**: Is there a git diff or file list available to review alongside the description?
- **CI status**: Did all automated checks pass (tests, lint, build)?
- **Links to verify**: Are there any external links in the PR description that should be checked?

Skip if answers are already available in the provided inputs.

### Step 3 — Produce the PR Check Report

Output a structured document using this exact format:

---

## PR Check Report: <PR title>

### Acceptance Criteria Review

| # | Criterion | Coverage | Notes |
| --- | --- | --- | --- |
| 1 | Description | ✅ Full / ⚠️ Partial / ❌ Missing | Explanation if not full |

**Coverage score**: N/M criteria fully covered.

### Definition of Done Review

| Item | Status | Notes |
| --- | --- | --- |
| Tests written and passing | ✅ / ⚠️ / ❌ | |
| Code reviewed and approved | ✅ / ⚠️ / ❌ | |
| Documentation updated | ✅ / ⚠️ / ❌ | |

### Scope Check

- **In scope and present**: Bulleted list of changes that correctly match the Technical Task.
- **Scope creep detected**: Bulleted list of changes NOT in the Technical Task. Each item must be flagged with a recommendation: accept / move to separate PR / revert.
- **In scope but missing**: Bulleted list of things the Technical Task required that are absent from the PR.

### Test Coverage Check

Based on the Testing Strategy from the Technical Task:
- **Unit tests**: ✅ Present / ⚠️ Partial / ❌ Missing — details
- **Integration tests**: ✅ Present / ⚠️ Partial / ❌ Missing — details
- **E2E / manual**: ✅ Covered in "How to Test" / ❌ Not described

### Link Check

If links are present in the PR description:
- List each link and its status: ✅ Reachable / ❌ Broken / ⚠️ Requires auth

If no links are present, write: "No links to verify."

### Open Technical Questions

List any Open Technical Questions from the Technical Task and whether the PR resolves, defers, or ignores each:

| Question | Status |
| --- | --- |
| Description | ✅ Resolved / ⏳ Deferred / ❌ Not addressed |

### Verdict

One of:
- **✅ Ready to merge** — all criteria covered, DoD met, no unresolved blockers.
- **⚠️ Merge with conditions** — minor issues listed; can merge if conditions are met.
- **❌ Needs rework** — blocking issues listed; must be addressed before merge.

### Blocking Issues
Numbered list. Only include items that must be fixed before merge. Empty if verdict is ✅.

### Non-blocking Suggestions
Bulleted list. Optional improvements for future PRs or follow-up tasks.

---

## Rules

- Never approve a PR that has uncovered acceptance criteria without an explicit justification in the PR description.
- Scope creep is not automatically bad — flag it and recommend, but do not auto-reject.
- Link checking is best-effort; if links require authentication or are internal, note that and skip.
- The verdict must be one of the three defined states — do not invent intermediate states.
- If the PR description was generated by `/pr-description`, cross-reference against the original Technical Task, not just the description.
- Write the report in the same language the user used.
